### 3.2 The "24-Hour COBOL Challenge" — POC Strategy

This is the core sales motion. It's irresistible because it costs the prospect nothing but time, and delivers a tangible artifact: working modern code from their own COBOL source.

**Script for the POC call:**

> "Here's the challenge: send us one real COBOL program from your production system. Not a toy example — a real program your team relies on. Within 24 hours, we'll return to you:
> 1. The equivalent Python/Java/Go code, fully commented
> 2. A generated test suite verifying the logic matches
> 3. An extracted business rules document
> 4. An API wrapper so you can call it from a modern stack
>
> If we can't deliver something you'd put into production, you owe us nothing. If we can, we'll talk about how to scale this across your entire estate."

**Why this works:**
- Zero risk for the buyer
- Creates "loss aversion" — they've seen it work on THEIR code
- Generates an internal champion (the developer who sent the file)
- Proves the product in the most credible way possible

**Operational Notes:**
- The 24-hour turnaround requires a human in the loop to validate the output
- This is fine — enterprise deals are worth £25K–£120K/year. Taking 4 hours to review a POC is a 100x ROI on time
- As the product improves, automate more of the validation

| Gap | Priority | Impact on Close Rate | Effort to Build | Note |
|-----|----------|---------------------|-----------------|------|
| **Mainframe connectivity** (CICS, DB2, VSAM, JCL support) | Critical | Blocking for 60%+ deals | High | Without CICS/VSAM/JCL parsing, you can't handle end-to-end mainframe programs. This is the #1 gap. |
| **Side-by-side regression testing** | Critical | Blocking for 40% | Medium | Enterprise needs proof that transpiled code produces identical output on real data. Ship a "comparison harness." |

| **EBCDIC → ASCII/UTF-8** | High | Required for mainframe data | Low | Simple encoding conversion but essential. |

| **JCL (Job Control Language) support** | High | Required for batch COBOL | Medium | JCL orchestrates COBOL batch jobs. Without it, batch programs are orphans. |
| **Security compliance documentation** | Medium | Required for banking | Medium | SOC 2, ISO 27001 attestation, data residency assurance. Banks won't upload source code without this. |
| **On-premise / air-gapped deployment** | Medium | Required for defense/gov | High | Government and defense won't upload sensitive COBOL to a cloud service. Ship a Docker image for on-prem. |

| **Audit trail / provenance** | Medium | Required for regulated industries | Medium | Show exactly which COBOL lines map to which output lines. Critical for audit sign-off. |

1. **Prioritize CICS+VSAM+COPYBOOK+EBCDIC** — Without these four, you cannot handle real mainframe COBOL. This is the difference between a "cool demo" and a revenue-generating product. The landing page promises "transpile COBOL" — but if you can only transpile standalone .cbl files without data access, you'll fail every enterprise POC.

2. **Build a regression test harness** — This is actually a sales tool as much as a product feature. The demo of "here's the original COBOL output, here's the Python output, they're identical" is the most powerful close you have.

3. **Write the security whitepaper** — A 2-page PDF explaining: "Your COBOL is processed in-memory, never stored, encrypted in transit, deleted after processing." This answers question #1 from every IT security team.
