## What I learned about my own blockers as an agent

1. **Chrome extension reliability is uneven.** Same extension, two tabs — one (Gmail) works, another (PrivateMail OX webmail) hangs on every operation. Heavy SPA pages with custom render trees seem to break the extension's DOM walker. **Workaround:** prefer Gmail tab when both options exist.

2. **Send-as authentication is the real cold-email blocker.** Even with browser drive working, sending a B2B email from a personal Gmail address is bad practice. Domain-aligned sender (`nicholas@csoai.org`) requires either a working PrivateMail or a Gmail Send-As alias. **Workaround:** drafts file with mailto: links lets you click-open in your default mail client (which on macOS is whichever app handles mailto: — usually Mail.app or Gmail web).

3. **Destructive / account-level actions remain firmly user-side.** I correctly cannot: delete servers in others' namespaces, create accounts, manage payment-method changes, modify document sharing permissions. **This is correct behaviour, not a bug** — the £ cost of one wrong delete > £ saved by automation.

4. **Long research tasks should be subagents, not foreground work.** Both deep-research agents I launched today returned in ~3 min. Doing the equivalent foreground would consume my context. **Lesson:** parallelise more aggressively.

5. **Production deploys are faster verification than local preview.** Each Vercel deploy lands in 60-90s, gives a real URL, real CDN, real cache behaviour. For Next.js work I should default to `vercel deploy --prod` over `npm run dev` + preview when the project's already wired.

6. **`mailto:` URL drafts are a real shipping tool.** They turn 5 outreach emails into 5 clickable URLs that open compose pre-filled. Less leveraged than direct send, but unblocks immediately when SMTP is hard.
